🟠 MAJOR — QA-006 — SSE `done` event never carries `contextId`
- **Boundary**: API↔FE / AI↔BE
- **Evidence**:
  - `backend/.../ai/rag/RAGChatbotEngine.java:131-134` emits `done` with data = `{"tokensIn":..,"tokensOut":..}` only.
  - `frontend/js/pages/chatbot.js:87-89` expects `payload.contextId` to update `activeContextId`.
- **Impact**: Every streamed message starts a fresh conversation because FE never captures the contextId. Multi-turn streaming chat is broken.
- **Responsible**: ai-engineer (emit contextId in `done` data) OR architect (clarify spec).
- **Fix direction**: Include `contextId` in the `done` JSON: `{"contextId":"...","tokensIn":..,"tokensOut":..}`. No FE change required; the parser already reads `payload.contextId`.

🟠 MAJOR — QA-007 — Default CORS origin list does not include all expected FE ports
- **Boundary**: API↔FE
- **Evidence**:
  - `CorsConfig.java:16` default `http://localhost,http://localhost:80,http://localhost:5173,http://localhost:3000`.
  - docker-compose does **not** set `CORS_ALLOWED_ORIGINS` env var — backend uses the default.
  - Architect spec said origins include `http://localhost:8080` and `https://makit.example.com`. Frontend dev also commonly runs on other ports.
- **Impact**: Accessing the app from the production FE container at `http://localhost` (port 80) works. Accessing from port 8080 or other will be blocked by CORS. Since same-origin via nginx is primary path, impact is low. But *any* direct-to-backend 8083 call from a static page served on 8080 will fail.
- **Responsible**: devops-engineer (add env var to compose) + architect (confirm final list).

🟠 MAJOR — QA-008 — No `Origin`/`withCredentials` tests; `allowCredentials=true` with explicit origin list is OK but `setAllowedHeaders(List.of("*"))` combined with `allowCredentials=true` is prohibited by the CORS spec for browsers
- **Boundary**: Security / API↔FE
- **Evidence**: `CorsConfig.java:23-25`. Spring validates `allowedHeaders=*` with `allowCredentials=true` → throws IllegalArgumentException on first request, OR Spring 6 rejects at bean init, OR silently downgrades — behavior version-dependent.
- **Impact**: Potential runtime CORS failures. Low if FE same-origin via nginx.
- **Fix direction**: Replace `List.of("*")` with explicit header list (`Authorization, Content-Type, X-Request-Id`).

🟡 MINOR — QA-M02 — `MarketingIntelligenceController.generateFeed` declares return type `ResponseEntity<?>`. Springdoc/OpenAPI cannot resolve the response schema for 200 vs 202.
- **Evidence**: `MarketingIntelligenceController.java:29`. Schema erased.

### Findings — Boundary 2

🔴 BLOCKER — QA-001 — `ContentGenerationStrategy` `NoUniqueBeanDefinitionException` at startup
- **Boundary**: AI↔BE
- **Evidence**:
  - 2 beans: `ClaudeTextContentStrategy` (`@Component("claudeTextContentStrategy")`) and `StableDiffusionImageStrategy` (`@Component("stableDiffusionImageStrategy")`). Neither is `@Primary`.
  - Services inject a *single* bean: `NlpAnalysisService.java:23` `private final ContentGenerationStrategy contentGen;`, same pattern in `FeedGenerationService.java:30`, `BackgroundRemovalService.java:27`, `ModelshotService.java:23`, `ReviewAnalysisService.java:26`, `UrlAnalysisService.java:17`, `YoutubeCommentsService.java:22`.
  - ai-engineer documentation (`_workspace/03_ai_progress.md:96-97`) explicitly says "inject as `List<ContentGenerationStrategy>` and pick by `.supports(type)`" — backend-engineer did NOT do this.
- **Impact**: Spring context fails to start. No endpoint serves any traffic.
- **Responsible**: backend-engineer (switch to `List<ContentGenerationStrategy>` injection + strategy pattern), OR architect (require `@Primary` on text strategy for simple path).
- **Fix direction**: In each service, inject `List<ContentGenerationStrategy> strategies` and pick by `.supports(type)`. Matches ai-engineer's contract.

🔴 BLOCKER — QA-002 — Duplicate `aiExecutor` bean definition (both `com.humanad.makit.config.AsyncConfig` and `com.humanad.makit.ai.config.AsyncConfig` declare `@Bean("aiExecutor")`)
- **Boundary**: AI↔BE / bean wiring
- **Evidence**:
  - `backend/.../config/AsyncConfig.java:12` — `@Bean("aiExecutor") public Executor aiExecutor()` — **NOT** `@ConditionalOnMissingBean`.
  - `backend/.../ai/config/AsyncConfig.java:24-25` — `@Bean(name="aiExecutor") @ConditionalOnMissingBean(name="aiExecutor") public AsyncTaskExecutor aiExecutor()`.
  - Both are `@Configuration`. Same simple class name (`AsyncConfig`) — Spring uses FQCN so distinct, but the two beans share the name `aiExecutor`.
  - `ConditionalOnMissingBean` only suppresses the AI one *if the backend one is seen first*. Order depends on component scan order (alphabetical by default): `com.humanad.makit.ai.config.AsyncConfig` would be scanned before `com.humanad.makit.config.AsyncConfig`. Result: AI's bean loads first, then backend's bean tries to register → `BeanDefinitionOverrideException` (default `spring.main.allow-bean-definition-overriding=false`).
- **Impact**: Context fails to start on exact scan order; even when it doesn't, the two differ in return type (`Executor` vs `AsyncTaskExecutor`) and `@Qualifier("aiExecutor") AsyncTaskExecutor` injections (in `ClaudeTextContentStrategy`/`StableDiffusionImageStrategy`) may only match the AI bean.
- **Responsible**: backend-engineer (delete or gate own `aiExecutor` with `@ConditionalOnMissingBean`) OR architect (decide which module owns it).
- **Fix direction**: Pick one location. If backend keeps the bean, return `AsyncTaskExecutor` not `Executor`, and mark `@ConditionalOnMissingBean` so tests can override.

🔴 BLOCKER — QA-003 — `BedrockProperties` keys not present in `application.yml`; `props.models().claudeHaiku()` will NPE on first AI call
- **Boundary**: AI↔BE / config
- **Evidence**:

  - `application.yml:48-55` only defines `aws.bedrock.text-model-id`, `aws.bedrock.image-model-id`, `aws.bedrock.enabled` — NOT the nested `models/defaults/tariff/s3` that the record expects.

- **Impact**: `models` record component is null → NPE at first invocation. Mock path does NOT use `props.models()` directly but the real service does.
- **Responsible**: backend-engineer (merge ai snippet into application.yml) OR devops-engineer (add to application-docker.yml).
